## 2024-07-26 - Optimize N+1 query loop with $in query and bulk insertMany
**Learning:** Found an N+1 query bottleneck in `src/controllers/workbasket.controller.js` where a `Team.findOne` was performed inside a loop over all PRIMARY workbaskets. This led to excessive DB round-trips. Furthermore, when a workbasket was missing, a separate `create` was fired inside the loop.
**Action:** Lift the query out of the loop and utilize the `$in` operator with a single `find` to check existence of multiple linked QC teams. Combine missing elements into a single `insertMany` to minimize network roundtrips to O(1) and improve scaling for firms with lots of workbaskets.
## 2024-05-18 - Replacing $facet with concurrent countDocuments()
**Learning:** While `$facet` aggregation pipelines are convenient for combining multiple count queries into a single database operation, they represent a significant anti-pattern for simple conditional counts on large collections. The initial `$match` requires MongoDB to load all matching documents into memory to evaluate the sub-pipelines, bypassing index scans and risking the 100MB aggregation memory limit.
**Action:** Always decompose `$facet` aggregations used for metric calculations into independent, concurrent `countDocuments()` queries executing within a `Promise.all()`. This allows MongoDB to resolve each count directly via index scans, yielding much faster and more scalable query execution.
